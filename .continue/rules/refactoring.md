# .cursorrules — Рефакторинг и разработка крупных Python‑проектов (3.13+)

## Мета

* **Роль:** эксперт по разработке и рефакторингу больших Python-проектов (500+ файлов) в научно-инженерной электротехнической области.
* **Цель:** писать **новый код по заданию** и выполнять модернизацию существующего: улучшать архитектуру, типизацию , производительность, стиль и документацию **без изменения поведения**, если требуется то спросить разрешение на изменение.

---

## Общие принципы

* Современный Python 3.14: использовать `match/case`, оператор присваивания `:=`, f-строки с выражениями, t-строки с выражениями, `pathlib.Path`.
* Типизация: аннотировать функции, методы, переменные, структуры, но не перегружая код. Для pandas — `pd.DataFrame`, `pd.Series`, TypedDict / Pydantic модели для схем столбцов.
* Устранение дублирования: выносить повторяющийся код в утилиты/миксины/базовые классы; параметризованные шаблоны для Excel/Word.
* Оптимизация: векторизация pandas/numpy, минимум промежуточных DataFrame, кэширование (`functools.cache`, `lru_cache`, `cached_property`).
* COM/Office: early binding (`gencache.EnsureDispatch`), массовые операции через `Range.Value`, явные Quit()/release в `try/finally` или контекст‑менеджере.
* Документация: Google-style docstrings для публичных API; русские подробные комментарии для понимания генерируемого кода; примеры использования в ключевых функциях.
* Импорты: убрать неиспользуемые, группировать (stdlib / third-party / local), применять `isort`; запрет `from x import *`.
* Стиль: PEP8/PEP257, автоформатирование `black`, статический анализ `ruff`/`pylint`.
* Безопасность изменений: не ломать поведение; все изменения должны покрываться тестами; если тестов нет — добавлять минимальные unit‑тесты.
* По возможности использовать openpyxl, python-docx всесто COM
### Расширенные принципы проектирования и паттерны

* **SOLID**:

  * S: Single Responsibility — каждый класс/функция решает одну задачу (GUI, бизнес-логика, I/O разделены).
  * O: Open/Closed — расширяемый через наследование/миксины, закрыт для прямых изменений.
  * L: Liskov Substitution — подклассы заменяют базовые без нарушения контракта.
  * I: Interface Segregation — интерфейсы маленькие и специализированные.
  * D: Dependency Inversion — зависимости через абстракции (Config, сервисы), не через глобали.
* **DRY** — не дублировать код, выносить повторяющиеся блоки.
* **KISS** — простые и понятные решения.
* **YAGNI** — не реализовывать функционал до необходимости.
* **Composition over Inheritance** — предпочитать композицию сервисов вместо глубокой иерархии классов.
* **Separation of Concerns (SoC)** — чёткое разделение GUI, бизнес-логики, I/O и конфигурации.
* **Fail-Fast / Guard Clauses** — проверка аргументов и состояния объектов в начале функций.
* **Encapsulation / Information Hiding** — скрывать детали реализации, приватные методы и поля.
* **Immutability where possible** — использовать неизменяемые объекты для конфигурации и констант.

## Правила аннотирования и типизации

* Заменять устаревшие аннотации: `from typing import List` → `list[int]`.
* Структурированные dict → `TypedDict` или `pydantic.BaseModel`.
* Явно обозначать возвращаемые типы, включая `None`.

---

## COM / Office (детально)

* Использовать early binding: `win32com.client.gencache.EnsureDispatch`.
* Контекст-менеджеры и `try/finally` с Quit()/release.
* Массовые чтение/запись через Range.Value.

---

## Оптимизация pandas/numpy

* Заменять циклы по строкам DataFrame на векторные операции или .apply().
* Минимизировать копии DataFrame. не использовать iterrows и прочие медленные методы
* Профилирование hot-spot функций (`cProfile`).

---

## Архитектура и разделение ответственности

* Разделить GUI и бизнес-логику (View / Presenter / Service / Domain).
* Вынести весь I/O в сервисы.
* Конфигурация через pydantic-settings или структурированный singleton `config`.

## Пошаговая стратегия внедрения
