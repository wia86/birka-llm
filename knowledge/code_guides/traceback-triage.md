# Traceback Triage

Правила разбора traceback, присланного пользователем Birka. Цель triage: быстро определить, на каком шаге пользовательского сценария произошел сбой, и дать безопасные рекомендации без раскрытия лишних внутренних деталей.

## Что извлекать из traceback

Извлекайте только минимум, нужный для диагностики:

1. Тип исключения.
2. Короткое сообщение ошибки.
3. Модуль или файл проекта, где произошел сбой.
4. Имя функции или метода.
5. Номер строки.
6. Последние 5-10 кадров стека.
7. Признак, что ошибка произошла:
   - при загрузке задания;
   - при проверке полей;
   - при запуске расчета;
   - в дочернем процессе;
   - при обработке результатов;
   - при формировании черновика отчета об ошибке.

## Что не нужно включать в ответ пользователю как есть

- абсолютные пути вида `C:\...`, `D:\...`, UNC-пути;
- данные профиля пользователя;
- токены, ключи, UUID лицензий;
- полный сырой лог;
- детали внутренней структуры, если они не помогают сделать безопасное действие в интерфейсе.

## Нормализация traceback перед анализом

Перед анализом:

1. Заменить абсолютные пути на нейтральные маркеры вида `<PATH>`.
2. Оставить только кадры, относящиеся к `birka`, если стек длинный.
3. Укоротить сообщение до сути ошибки.
4. Проверить, нет ли в тексте путей к входным данным, сетевых папок и персональных имен.

## Как сопоставлять traceback со сценарием пользователя

### Ошибка при загрузке задания
Признаки:
- упоминание загрузки YAML или файла задания;
- проблема возникает сразу после открытия `.calc`, `.edit`, `.static`, `.tkz`, `.other`.

Что сказать пользователю:
- программа не смогла корректно прочитать или восстановить задание;
- нужно проверить формат файла задания и актуальность сохраненных путей.

### Ошибка при проверке данных перед запуском
Признаки:
- ошибка возникает до старта расчета;
- пользователь видит предупреждение или отказ до появления результатов.

Что сказать пользователю:
- проблема в заполнении полей, выборе панелей, источнике данных или конфликте настроек.

### Ошибка во время расчета
Признаки:
- запуск начался, но завершился с исключением;
- есть кадры из блока выполнения расчета;
- ошибка может быть общей или в дочернем процессе.

Что сказать пользователю:
- запуск прервался на этапе обработки данных или генерации результатов;
- сначала нужно проверить входные данные и настройки сценария.

### Ошибка в дочернем процессе
Признаки:
- traceback указывает на обработку отдельной модели или процесса;
- сбой может быть связан только с частью набора данных.

Что сказать пользователю:
- проблема может быть связана с конкретным файлом РМ или конкретным элементом пакетной обработки;
- полезно проверить воспроизводимость на одном файле.

## Шаблон диагностики для ответа пользователю

Используйте такой формат:

1. Что означает ошибка простыми словами.
2. На каком шаге сценария она возникает.
3. Что проверить в данных задания и в настройках.
4. Какие безопасные действия выполнить для исправления.
5. Когда нужно эскалировать проблему разработчику.

### Готовый шаблон ответа

1. `Что означает ошибка:` программа не смогла завершить шаг `<название шага>` из-за `<короткая причина простыми словами>`.
2. `Где возникает:` ошибка появляется при `<загрузка задания / проверка / расчет / пакетная обработка / формирование результата>`.
3. `Что проверить:` источник РМ, заполненность обязательных полей, порядок панелей, корректность файла задания, внешние зависимости для выбранного результата.
4. `Что сделать:` переуказать входной файл или папку, отключить лишние панели, проверить обязательные поля, повторить запуск на одном файле, скорректировать параметры панели.
5. `Когда эскалировать:` если ошибка повторяется на корректном задании после исправления входных данных.

## Матрица "Ошибка -> Причина -> Проверка -> Исправление -> Профилактика"

| Ошибка | Причина | Проверка | Исправление | Профилактика |
|---|---|---|---|---|
| `KeyError` или похожая ошибка выбора панели/опции | Несогласованность состава панелей, устаревшее задание, недоступная опция | Проверить модуль, состав панелей, корректность загруженного задания | Пересоздать проблемную панель или сохранить задание заново | Использовать актуальные шаблоны заданий |
| Ошибка чтения файла | Неверный путь, формат, кодировка или недоступность файла | Проверить путь и тип файла | Переуказать файл, пересохранить задание | Не использовать старые абсолютные пути без проверки |
| Ошибка пакетной обработки | Один проблемный файл в наборе | Повторить сценарий на одном файле | Исключить проблемный файл и локализовать проблему | Сначала проверять сценарий на эталонном файле |
| Ошибка внешней зависимости | Не настроен внешний компонент или он недоступен | Проверить настройки внешнего инструмента | Настроить зависимость или отключить соответствующий результат | Периодически проверять окружение после обновлений |

## Как переводить traceback в язык интерфейса

Не отвечайте пользователю фразами вроде:
- "сломался метод";
- "ошибка в слое infrastructure";
- "падает в дочернем пайплайне".

Лучше отвечать так:
- "программа не смогла загрузить файл задания";
- "не удалось проверить выбранные панели перед запуском";
- "ошибка возникла при обработке одной из моделей";
- "не удалось сформировать рисунки/таблицы/пояснительную записку".

## Когда просить дополнительные материалы

Попросить пользователя прислать traceback или лог дополнительно, если:
- в сообщении недостаточно данных для определения этапа сбоя;
- ошибка воспроизводится, но непонятно, на каком входном файле;
- есть подозрение, что проблема в конкретном файле задания или конкретной модели.

Попросить файл задания дополнительно, если:
- ошибка явно связана с загрузкой или настройкой сценария;
- нужно проверить состав панелей и тип источника данных.

## Когда эскалировать разработчику

Эскалируйте, если:
- ошибка воспроизводится на корректном сценарии после исправления данных;
- ошибка началась после обновления версии;
- сбой происходит в разных сценариях при одинаковых шагах пользователя;
- в traceback нет явной пользовательской причины.
