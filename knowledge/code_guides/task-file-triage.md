# Task File Triage

Правила безопасного разбора загруженного файла задания на расчет. Цель triage: извлечь только те признаки, которые помогут поддержке понять сценарий пользователя и дать практические рекомендации.

## Какие форматы задания поддерживает программа

Основной формат задания:
- YAML-файл с расширением модуля: `.calc`, `.edit`, `.static`, `.tkz`, `.other`.

Особенности:
- файл может быть сохранен в UTF-8;
- старые задания могут встречаться в CP1251;
- при загрузке могут автоматически мигрироваться устаревшие названия панелей и структура старого формата.

## Что безопасно извлекать из файла

Извлекать только минимальный набор признаков:

1. Версия формата или версии приложения, если она присутствует.
2. Тип модуля (`calc`, `edit`, `static`, `tkz`, `other`).
3. Тип источника данных:
   - файл;
   - папка;
   - папка со вложенными папками.
4. Признак, включены ли исключения для пакетной обработки.
5. Состав панелей:
   - какие панели есть;
   - какие панели отключены;
   - есть ли дубли панелей результатов.
6. Ключевые параметры сценария:
   - есть ли контролируемые элементы;
   - есть ли отключаемые элементы;
   - какие блоки результатов включены;
   - есть ли панели ТКЗ;
   - есть ли пакетная обработка.
7. Диагностические признаки:
   - пустой источник;
   - отсутствующие обязательные секции;
   - конфликт порядка панелей;
   - неподдерживаемые расширения файлов;
   - старый формат задания.

## Что не следует извлекать или сохранять для RAG

- абсолютные пути к файлам и каталогам;
- персональные или сетевые пути пользователя;
- полный раздел настроек пользователя;
- содержимое логов, приложенных к заданию;
- лишние данные, не связанные с диагностикой.

## Базовая структура файла задания

Техподдержке полезно ориентироваться на следующие логические разделы:

- `version`
- `name_module`
- `created_at`
- `source`
- `panels`
- `settings`

Если один из ключевых разделов отсутствует, задание считается подозрительным и требует дополнительной проверки.

## Минимальный набор извлекаемых признаков

### Версия формата
- строка версии;
- наличие признаков старого формата;
- необходимость миграции имен панелей.

### Тип расчета
- модуль;
- цель сценария по составу панелей.

### Ключевые параметры
- наличие и тип источника РМ;
- наличие исключений;
- наличие контролируемых элементов;
- наличие отключаемых элементов;
- активные панели результата;
- выбор формата результата, если он влияет на диагностику.

### Состав схемы и узлов
Из файла задания безопасно извлекается не сама схема, а только признаки:
- используется ли выборка узлов;
- используется ли пакетный источник;
- есть ли сценарии по оборудованию ТКЗ;
- есть ли сценарии по сети, графу, конвертации или пояснительной записке.

### Диагностические сообщения
Если файл задания явно указывает на некорректную или неполную структуру, это нужно фиксировать как диагностический сигнал.

## Правила валидации файла

### Файл считается корректным, если
- расширение соответствует модулю;
- YAML читается без ошибки;
- есть ключевые разделы;
- модуль определен однозначно;
- состав панелей соответствует модулю;
- обязательные секции заполнены хотя бы на минимальном уровне.

### Файл считается некорректным или подозрительным, если
- не читается как YAML;
- нет `name_module` или `panels`;
- источник данных пустой;
- в `panels` нет полезных включенных панелей;
- есть конфликтующие или дублирующиеся панели результата;
- для `tkz` нарушен порядок ключевых панелей;
- файл относится к другому модулю, чем ожидает пользователь;
- задание содержит устаревшие или частично мигрированные данные.

## Типовые причины отказа расчета по файлу задания

1. Пустой или устаревший путь к источнику данных.
2. Обязательные панели не настроены.
3. Панель результата включена, но ее входные данные не заданы.
4. В файле задания сохранены параметры для другой машины или другого окружения.
5. Состав панелей противоречит сценарию пользователя.

## Алгоритм triage по файлу задания

1. Определить модуль по расширению и внутреннему полю модуля.
2. Проверить, что файл читается как YAML.
3. Выделить безопасные метаданные: версия, тип модуля, тип источника, наличие исключений.
4. Составить краткий список панелей:
   - включенные;
   - отключенные;
   - подозрительные.
5. Проверить обязательные условия сценария:
   - источник РМ;
   - наличие результата;
   - наличие обязательных исходных данных;
   - порядок панелей для ТКЗ.
6. Подготовить ответ пользователю в терминах интерфейса.

## Шаблон ответа пользователю по файлу задания

1. Что удалось распознать из файла.
2. Какие несоответствия или ошибки найдены.
3. Какие поля или разделы нужно поправить.
4. Какие шаги выполнить в интерфейсе программы.
5. Когда нужно дополнительно прислать traceback или лог.

### Готовый шаблон

1. `Что удалось распознать:` файл задания относится к модулю `<модуль>`, использует `<тип источника>`, содержит `<ключевые панели>`.
2. `Что найдено:` обнаружены `<проблемы или несоответствия>`.
3. `Что поправить:` проверьте `<источник / конкретные панели / обязательные поля / порядок панелей>`.
4. `Что сделать в интерфейсе:` откройте модуль `<модуль>`, загрузите задание, исправьте перечисленные поля и повторите запуск.
5. `Что прислать дополнительно:` если после исправлений ошибка повторится, пришлите traceback или короткий лог запуска.

## Таблица соответствия "сигнал из файла -> вероятная причина -> действие пользователя"

| Сигнал из файла задания | Вероятная причина | Действие пользователя |
|---|---|---|
| Пустой `source` | Источник РМ не задан или потерян при переносе задания | Заново выбрать файл или папку в блоке `Выбор моделей` |
| Указан путь, но он явно машинозависимый | Задание создано на другой машине | Переуказать источник данных вручную |
| Нет включенных панелей результата | Пользователь сохранил неполный сценарий | Добавить и включить нужную панель результата |
| Есть панель результата без исходной панели | Настройка сценария неполная | Добавить недостающую панель исходных данных |
| Повтор одной и той же панели результата | Задание сохранено с конфликтом | Оставить одну копию панели, остальные отключить |
| Для `tkz` панель расчета стоит раньше панели оборудования | Некорректный порядок панелей | Переставить панели в рабочей области |
| Есть старые названия панелей | Старый формат задания | Пересохранить задание в текущей версии |
| Неподходящее расширение файла для сценария | Пользователь открыл файл не тем способом | Открыть файл через подходящий модуль или сценарий |
| Нет обязательных секций YAML | Поврежденный или вручную испорченный файл | Пересоздать файл задания из программы |
| Есть признаки пакетной обработки и исключений | Пользователь запускает набор моделей | Проверить, не исключены ли нужные файлы и папки |

## Когда просить traceback или лог дополнительно

Попросить traceback или лог, если:
- структура файла выглядит корректной, но запуск все равно падает;
- проблема проявляется только на выполнении, а не при загрузке;
- нужно локализовать сбой на конкретном шаге после чтения задания.

## Когда достаточно только файла задания

Файла задания обычно достаточно, если:
- он не загружается;
- загружается не в тот модуль;
- пропадают панели;
- очевидны пустые обязательные поля;
- нарушен порядок панелей или состав сценария.
