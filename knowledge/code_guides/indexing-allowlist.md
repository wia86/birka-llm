# Indexing Allowlist

Правила отбора материалов для индексации в RAG по проекту Birka. Цель allowlist: индексировать только curated knowledge и безопасные пользовательские описания.

## Включать в индекс

### Основной набор
- `knowledge/code_guides/features-overview.md`
- `knowledge/code_guides/workflows.md`
- `knowledge/code_guides/errors-and-fixes.md`
- `knowledge/code_guides/traceback-triage.md`
- `knowledge/code_guides/task-file-triage.md`
- `knowledge/code_guides/limits-and-assumptions.md`
- `knowledge/code_guides/glossary.md`
- `knowledge/code_guides/indexing-allowlist.md`

### Допустимые дополнительные источники
- пользовательская документация в `docs`, если документ прошел safety-check;
- обезличенные примеры файлов заданий, специально подготовленные для обучения поддержки;
- краткие справочные материалы по интерфейсу без внутренних путей и секретов.

## Условно допустимо, но только после ручной проверки

- выдержки из кода, преобразованные в markdown-описания;
- фрагменты traceback после маскировки путей и чувствительных данных;
- обезличенные логи, если они нужны для типовых сценариев поддержки.

## Исключать из индекса

### По директориям проекта
- `src/birka/infrastructure/config`
- `src/birka/infrastructure/licensing`
- `src/birka/infrastructure/enter`
- `src/birka/infrastructure/rastr`
- `src/birka/core`
- любые тестовые артефакты, временные файлы, кэши и логи

### По типам содержимого
- секреты, ключи, токены, UUID лицензий;
- внутренние URL, корпоративные эндпоинты, служебные e-mail;
- абсолютные локальные и сетевые пути;
- персональные данные пользователя;
- сырые файлы настроек пользователя;
- полные traceback и логи без маскировки.

## Denylist признаков чувствительных данных

Перед индексацией проверять markdown на наличие:

- путей вида `C:\`, `D:\`, `\\server\share\`;
- строк с `token`, `secret`, `apikey`, `api_key`, `password`, `uuid`, `license`;
- e-mail-адресов, если они не являются публичными служебными адресами;
- внутренних URL и адресов инфраструктуры;
- длинных сырых фрагментов логов;
- ссылок на локальные профили пользователей.

## Прединдексационный чеклист

1. Документ относится к пользовательскому сценарию или triage, а не к внутренней реализации.
2. В документе нет абсолютных путей и персональных данных.
3. В документе нет ключей, токенов, лицензий и внутренних URL.
4. Термины приведены к языку интерфейса и действий пользователя.
5. Для traceback и файлов заданий используются только обезличенные признаки.
6. Документ понятен инженеру техподдержки без чтения исходного кода.

## Правило принятия решения

Если документ вызывает сомнение по приватности или слишком привязан к внутренней реализации, не добавлять его в индекс до ручной проверки и очистки.
